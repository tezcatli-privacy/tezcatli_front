# Tezcatli Web

Frontend web for the Tezcatli alpha. This app is the user-facing shell that connects four pieces of the product:

- public landing page
- wallet privacy scan flow
- migration and confidential smart account flow
- auditor surface for operational compliance visibility

The frontend is intentionally thin in business logic. It renders the alpha flow, talks to `tezcatli_back` for scan and migration eligibility, talks to `tezcatli_compliance` for the auditor surface, and reads the shared Arbitrum Sepolia deployment manifest generated by the contract repos.

## What This Repo Owns

This repo owns the web UI and client-side transaction orchestration for the alpha.

Main surfaces:

- `/` landing page
- `/scan` wallet app for scan, eligibility, migration, smart account, and private DeFi actions
- `/audit` auditor dashboard backed by compliance APIs

Core frontend responsibilities:

- connect an injected wallet
- run the privacy scan against `tezcatli_back`
- request migration eligibility separately from the privacy scan
- display supported alpha assets and migration state
- submit onchain actions for:
  - wrapped asset migration
  - Permit2 batch migration
  - confidential smart account usage
  - confidential vault deposit
- show auditor metrics from `tezcatli_compliance`

This repo does **not** own:

- privacy scoring connectors and scan orchestration
- compliance policy logic or identity linking
- contract deployment and wiring

Those live in the sibling repos:

- `../tezcatli_back`
- `../tezcatli_compliance`
- `../tezcatli_migrator`

## Alpha Flow

The current alpha flow is intentionally split into three product phases:

1. `Scan Wallet`
   The user connects a wallet and requests a privacy scan. This step uses `POST /api/scan` from `tezcatli_back` and returns privacy score, findings, recommendations, supported assets, and next actions.

2. `Request migration eligibility`
   This is a separate decision point. The frontend calls `POST /api/migration/eligibility` only when the user explicitly asks to migrate. For alpha and demo work, this path may use simulated Wavy-aligned risk bands so the rest of the flow can be exercised without blocking on the live provider.

3. `Create confidential account and use DeFi confidentially`
   If the wallet is eligible, the app prepares the confidential smart account, migrates selected assets, shows the resulting account, and exposes the next alpha actions:
   - generate private yield
   - buy gold privately

## How The Frontend Is Wired

### 1. Backend APIs

`tezcatli_back` is the source of truth for the user scan flow.

Used endpoints:

- `POST /api/scan`
- `GET /api/scan/:scanSessionId/status`
- `GET /api/scan/:scanSessionId/report`
- `POST /api/migration/eligibility`
- recovery and remediation endpoints when enabled

### 2. Compliance APIs

`tezcatli_compliance` currently backs the auditor surface.

Used endpoints:

- `GET /v1/compliance/audit/summary`
- `GET /v1/compliance/audit/events`
- `GET /v1/compliance/audit/cases`

### 3. Shared Deployment Manifest

The frontend loads:

- [`../shared/deployments/arbitrum-sepolia.alpha.json`](../shared/deployments/arbitrum-sepolia.alpha.json)

That manifest is treated as the source of truth for:

- migrator addresses
- 4337 account factory
- paymaster
- vaults
- wrapped assets
- GMX adapter
- compliance gate references
- active USDC strategy lanes

The loader lives in [`lib/alpha-manifest.ts`](./lib/alpha-manifest.ts).

## Local Development

### 1. Use Node 22

```bash
nvm use 22
```

### 2. Install dependencies

```bash
npm install
```

### 3. Configure environment

Copy `.env.example` to `.env.local` and adjust if needed.

Current variables:

```bash
NEXT_PUBLIC_TEZCATLI_BACKEND_URL=http://localhost:3001
NEXT_PUBLIC_TEZCATLI_COMPLIANCE_URL=http://localhost:4100
NEXT_PUBLIC_ARBITRUM_SEPOLIA_RPC_URL=https://sepolia-rollup.arbitrum.io/rpc
```

Notes:

- `tezcatli_back` usually runs on `3001`
- `tezcatli_compliance` usually runs on `4100`
- if you are pointing at a different RPC provider, update `NEXT_PUBLIC_ARBITRUM_SEPOLIA_RPC_URL`

### 4. Start the app

```bash
npm run dev
```

Default URL:

- [http://localhost:3000](http://localhost:3000)

## Demo Notes

The current alpha frontend contains demo-oriented behavior to keep the end-to-end flow testable:

- migration eligibility can be simulated in backend alpha mode
- demo wallets can be prelinked in compliance so the migration lane stays operable
- the frontend uses a deterministic migration salt per wallet so demo smart accounts can be predicted and prelinked

That means the UI is already shaped for product validation, but some pieces are still alpha-grade:

- live Wavy dependency is not always required during local demo work
- buy-gold UX is present, but the exact backend route or adapter semantics may still evolve
- compliance behavior in demo mode may be intentionally softer than final production policy

## Current Intent

This frontend is not trying to be a generic wallet dashboard. It is the alpha shell for a very specific story:

- understand current exposure
- decide if migration is allowed
- move into a confidential smart account
- use private DeFi primitives
- give the auditor a separate operational view

When updating the UI, keep that sequencing intact and avoid pushing policy logic that should stay in backend, compliance, or the shared manifest.
